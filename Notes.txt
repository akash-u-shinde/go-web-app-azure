				kubectl port-forward creates a temporary tunnel from my local machine to a Kubernetes service or pod. It allows me to access the application locally without 				exposing it externally. For example, I map localhost port 8080 to service port 80, which forwards traffic to the running pod.
				It is mainly used for debugging and development, as it bypasses Kubernetes networking and directly connects to the resource through the API server.

				Local Port → kubectl tunnel → Service → Pod

# CI using Github Actions

name: CI

# Exclude the workflow to run on changes to helm chart

on:
  push:
    branches:
      - main
    paths-ignore:
      - 'helm/**'
      - 'k8s/**'
      - 'Readme.md'

permissions:
  id-token: write   # 🔥 REQUIRED
  contents: write
  
jobs:

  build: # This is the name of the job1
    runs-on: ubuntu-latest

    steps:

    - name: Checkout Repository
      uses: actions/checkout@v4

    - name: Set up Go 1.22.5
      uses: actions/setup-go@v2
      with:
        go-version: 1.22.5

    - name: Build
      run: go build -o go-web-app 

    - name: Test
      run: go test ./...

  code-quality: # This is the name of the job2.
    runs-on: ubuntu-latest

    steps:

    - name: Cheout Repository
      uses: actions/checkout@v4

    - name: Set up Go 1.22.5
      uses: actions/setup-go@v2
      with:
        go-version: 1.22.5

    - name: Run golangci-lint
      uses: golangci/golangci-lint-action@v6
      with:
        version: v1.56.2


  push: #This is name of the job3
    runs-on: ubuntu-latest
    needs: [build, code-quality] # This job will run after the build and code-quality jobs have completed successfully.

    steps:
    - name: Checkout Repository
      uses: actions/checkout@v4

    - name: Azure Login (OIDC)
      uses: azure/login@v2
      with:
        client-id: ${{ secrets.AZURE_CLIENT_ID }}
        tenant-id: ${{ secrets.AZURE_TENANT_ID }}
        subscription-id: ${{ secrets.AZURE_SUBSCRIPTION_ID }}

    - name: Login to ACR
      run: az acr login --name ${{ secrets.ACR_NAME }}

    - name: Build Image
      run: |
        docker build -t ${{ secrets.ACR_NAME }}.azurecr.io/go-web-app:${{ github.run_id }} .

    - name: Push Image
      run: |
        docker push ${{ secrets.ACR_NAME }}.azurecr.io/go-web-app:${{ github.run_id }}


  update-helm:
    runs-on: ubuntu-latest
    needs: push

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4
        with:
          token: ${{ secrets.TOKEN }}

      - name: Update tag in Helm chart
        run: |
          sed -i 's/tag:.*/tag: "${{ github.run_id }}"/' helm/go-web-app-chart/values.yaml

      - run: |
          git config --global user.email "akash.shinde1808@gmail.com"
          git config --global user.name "Akash Shinde"
          git add helm/go-web-app-chart/values.yaml
          git commit -m "Update tag in Helm chart"
          git push

🚀 🔐 COMPLETE ACCESS SUMMARY FOR YOUR CI PIPELINE

🧠 1. GITHUB ACTIONS PERMISSIONS (VERY IMPORTANT)
👉 In your workflow YAML:

permissions:
  contents: write     # 🔥 required for git push (Helm update)
  id-token: write     # 🔥 required for Azure OIDC login

🎯 Why?
Permission	Purpose
contents: write	Push code changes to repo
id-token: write	Generate OIDC token for Azure login

Add in repo → Settings → Secrets:

AZURE_CLIENT_ID
AZURE_TENANT_ID
AZURE_SUBSCRIPTION_ID
DOCKERHUB_USERNAME   (if using DockerHub)
DOCKERHUB_TOKEN
TOKEN (PAT for git push if used)

	• Create Azure AD App

🔹 Federated Credential (OIDC)

repo:akash-u-shinde/go-web-app-azure:ref:refs/heads/main

🔹 RBAC ROLE (MANDATORY 🔥)
👉 Assign role at Subscription level
Role	Purpose
Contributor	Full access to deploy resources

🔹 DockerHub

docker login

DOCKERHUB_USERNAME

Role	Purpose

   ↓ (permissions: id-token, contents)

Azure Subscription

	To run the CI/CD pipeline, I require GitHub permissions like id-token: write for OIDC authentication and contents: write for pushing changes. In Azure, I configure an App Registration with federated credentials and assign a Contributor role at the subscription level. Additionally, I provide access to the container registry for image push and AKS permissions for deployment.

💥 PRO LEVEL ANSWER
	I follow a least-privilege model, granting only necessary permissions such as AcrPush for registry access and scoped RBAC roles for Kubernetes deployments.
